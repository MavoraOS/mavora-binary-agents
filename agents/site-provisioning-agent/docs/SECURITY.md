# SECURITY — Mavora Site Provisioning Agent

SP-T-219 / agent/07 §42. Trust model, threat model, and security operations.
Model C: agent is outbound HTTPS-443 only. AOP/SSL are out of scope.

---

## Trust Model — 3-Factor Channel Auth (SP-ADR-007)

Every request from the agent to Mavora carries all three factors:

| Factor | How | Notes |
|---|---|---|
| mTLS client certificate | Mutual TLS on every HTTP call | Cert signed by Mavora internal CA; private key generated on-VPS and NEVER transmitted |
| Egress IP | Server-side IP check | Must match the IP registered in the agent record on Mavora |
| Bearer token (`connection_key`) | Authorization header | Received from control-plane at bootstrap; loaded in-memory only; NEVER logged |

All three factors must pass on every `/agent/v1/*` call. Failure on any = 401/403.

---

## Ed25519 Signed Job Envelopes (SP-ADR-007)

Every job dispatched to the agent is signed by Mavora with an Ed25519 private key.

- **Agent holds the Ed25519 VERIFYING PUBLIC KEY only** — the signing private key stays at Mavora.
- The agent cannot forge jobs even if a Customer with root access reads the config files.
- **Blast radius = 1 VPS** — revoking one agent (bearer + cert + Ed25519 pub) does not affect others.

### Envelope Verification Steps (SP-T-202)

1. Ed25519 signature over canonical bytes `{job_id, agent_id, lease_token, idempotency_key, issued_at, nonce, command_type, payload, signature_key_id}`
2. `issued_at` within ±5 minutes of current time
3. Nonce and job_id not previously seen (replay seen-set)
4. `command_type` in allowlist (`provision_site`, `deprovision_site`)
5. Payload schema validates strictly

Any failure → job rejected before execution (`AGENT_SIGNATURE_INVALID` or `AGENT_REPLAY_DETECTED`).

## Install Token — Single-Use, Short-Lived

- Generated by Mavora UI when the agent record is created.
- Valid for 1 hour; consumed on first successful bootstrap exchange.
- Must not be stored on disk longer than needed (write to `/tmp`, chmod 0600, delete after install).
- Use `--token-file` flag to load from file (keeps token off shell history).
- After bootstrap, the token is discarded; credentials (cert + bearer + Ed25519 pub) are used instead.

---

## mTLS Private Key — Never Transits

The mTLS keypair is generated on the VPS during `install.sh`:

```
openssl ecparam -name prime256v1 -genkey -noout -out /etc/mavora-agent/client.key
```

Only the CSR (Certificate Signing Request, which contains the PUBLIC key) is sent to Mavora during bootstrap. The private key is stored in `/etc/mavora-agent/client.key` (0600, owner mavora-agent) and NEVER leaves the VPS.

Mavora signs the CSR and returns the signed certificate. Before starting the agent, `install.sh` verifies that the returned certificate matches the on-disk private key (`cert↔key` check). If they do not match, the agent is NOT started.

## Secret File Permissions (RC-7)

| File | Mode | Why |
|---|---|---|
| `/etc/mavora-agent/` | 0700 | No world-read on config directory |
| `client.key`, `client.crt`, `bearer.token`, `signing.pub`, `config.yaml` | 0600 | Secrets — only mavora-agent process can read |
| `ca.crt` | 0644 | CA cert is not a secret |

**Agent refuses to start** if any secret file is not 0600 or not owned by mavora-agent.
Error code: `AGENT_SECRET_PERMISSION_UNSAFE`.

Backup tooling must exclude `/etc/mavora-agent/` from plaintext backups (RC-7).

## systemd Hardening (SP-BR-18)

The unit file enforces:

- `User=mavora-agent` — least-privilege process user (no root)
- `NoNewPrivileges=yes` — cannot escalate via suid/capability (sudo policy still applies)
- `ProtectSystem=strict` — filesystem read-only except `ReadWritePaths`
- `ProtectHome=yes` — no /home or /root access
- `PrivateTmp=yes` — isolated /tmp
- `CapabilityBoundingSet=` empty — all capabilities dropped; sudo provides allowlisted ops
- `RestrictSUIDSGID=yes` — cannot set SUID/SGID on files
- `LockPersonality=yes` — cannot change execution domain
- `ProtectKernelTunables=yes` / `ProtectKernelModules=yes` / `ProtectControlGroups=yes`

`MemoryDenyWriteExecute` is intentionally OMITTED because WP-CLI PHP/JIT may require W+X memory. See `deploy/systemd/mavora-agent.service` comments.

## Sudoers Allowlist (Narrow, No Wildcard)

`/etc/sudoers.d/mavora-agent` allows only four specific commands:

| Command | Purpose |
|---|---|
| `/usr/sbin/nginx -t` | Validate nginx config (read-only) |
| `/bin/systemctl reload nginx` | Safe reload (no restart, no arbitrary service) |
| `/opt/mavora-agent/bin/write-vhost` | Write nginx vhost via root-owned helper |
| `/opt/mavora-agent/bin/wp-safe` | WP-CLI via allowlisted helper (drops to www-data) |

**No `ALL=(ALL)`, no wildcards, no `systemctl *`.**
The sudoers file is validated with `visudo -c` before installation.

Firewall entry is commented by default (only uncomment when `firewall_enabled=true`, SP-BR-17).

## Credential Rotation (~90 Days)

Rotate proactively every ~90 days or immediately on suspected compromise:

```bash
# From Mavora UI: generate a rotate token (Agents → [Agent] → Rotate)
sudo bash scripts/rotate.sh \
  --control-plane https://api.mavora.io \
  --token-file /tmp/rotate-token
```

### What rotation does:
1. Generates NEW mTLS keypair + CSR on this VPS (private key stays here).
2. Exchanges rotate token + CSR for: new signed cert + new bearer + new Ed25519 public key.
3. Verifies new cert ↔ new key.
4. Installs new credentials (0600).
5. Reloads service.
6. Keeps **both** old + new Ed25519 public keys for a **grace window** (default 15 minutes).
7. After grace: removes old Ed25519 public key; only new key accepted.

### Grace window rationale:
Mavora may send jobs signed with the old signing key while the rotate is in progress.
The agent accepts both old and new signatures during grace, preventing dropped jobs.

## BYO-VPS Threat Model (SP-ADR-007 / master-spec §27)

The Customer has root access to their own VPS. This is the intended deployment model.

Security properties even with Customer root access:
- **Cannot forge jobs**: Ed25519 signing private key is at Mavora only.
- **Cannot impersonate the agent**: bearer + mTLS cert tied to one agent record; compromised cert → revoke on UI → blast=1 VPS.
- **Cannot read other agents**: per-agent credentials (separate bearer, cert, Ed25519 pub per agent).
- **Cannot pivot to other VPS**: agent does not SSH to other machines; outbound 443 only.

If a VPS is suspected compromised: revoke on Mavora UI → rebuild VPS → reinstall agent.

## NEVER LOG List (RC-7 §38.2)

The structured log scrubber (`internal/logging/scrubber.go`) redacts any field matching `*key*|*token*|*password*|*secret*|authorization|credential*`.

These values MUST NEVER appear in log output:

- `connection_key` (bearer token)
- mTLS private key or CSR private material
- WP Application Password (plaintext — passed to backend in-memory then discarded)
- Authorization header (raw)
- Ed25519 signing material
- `install_token` (in installer logs)

## Optional Firewall (SP-BR-17)

Default: **OFF** (`firewall_enabled: false`).

When enabled, the agent applies an IP-allowlist on inbound port 80 for known proxy CIDRs. Provisioning does NOT fail if this step fails (`AGENT_FIREWALL_LOCK_FAILED` is a side-code, not a gate).

AOP (Application Origin Protection), Let's Encrypt, and Cloudflare are **not configured by the agent** (Model C: Customer manages SSL and DNS).

---

## AOP/SSL — Out of Scope (Model C)

The agent serves sites via HTTP port 80 only (origin). SSL termination is the Customer's responsibility. The agent does NOT:
- Configure Let's Encrypt / certbot
- Manage Cloudflare or any CDN
- Touch DNS records
- Configure nginx SSL (`listen 443 ssl`, `ssl_certificate`, etc.)
